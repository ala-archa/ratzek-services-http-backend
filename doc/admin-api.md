# Админка: API-контракт для фронтенда

Инструкция для разработки админ-панели с нуля. Покрывает аутентификацию
(login/logout) и CRUD управления «безлимитными» (no-shaping) IP-клиентами.

Бэкенд — Rust/Actix, слушает `127.0.0.1:8888` за обратным прокси. Все ответы —
`HTTP/1.1`, тела запросов/ответов — JSON, если не указано иное.

---

## 0. Базовые правила

- **База на проде:** **`http://www.ratzek/api/...`** — внутренний хост café-LAN за
  nginx, **только HTTP, без TLS**. Бэкенд внутренне слушает `127.0.0.1:8888`, наружу
  отдаётся через `location /api/ → proxy_pass http://127.0.0.1:8888/api/`. Все
  эндпоинты — под `/api/v1/...` (т.е. `http://www.ratzek/api/v1/admin/...`).
- **Same-origin + куки.** Сессия передаётся в **HttpOnly cookie** `admin_session`
  (JS её прочитать не может). Поэтому:
  - Админка обслуживается с того же origin по адресу **`http://www.ratzek/ap-admin/`**.
    nginx уже настроен: `location /ap-admin/ → alias /var/www/ratzek-ap-admin/` c
    SPA-fallback на `index.html` (любой под-маршрут отдаёт `index.html`), плюс редирект
    `/ap-admin` → `/ap-admin/`. CORS на бэкенде нет, поэтому **только same-origin**.
  - В каждом `fetch` указывай `credentials: 'include'` (или `'same-origin'`),
    иначе куку не отправит и получишь `401`.

> **Деплой админки:** собирай SPA с **base path `/ap-admin/`** (Vite: `base: '/ap-admin/'`,
> CRA: `homepage`), выкладывай билд в `/var/www/ratzek-ap-admin/` (напр. `rsync`).
> API дёргай по **абсолютным** путям `/api/v1/...` (тот же origin) — отдельный прокси
> не нужен, `location /api/` уже проксирует на бэкенд.
- **Куку не трогаем руками.** Браузер сам её хранит и шлёт. Признак «залогинен» —
  успешный `GET /api/v1/admin/me`, а не наличие куки.
- **Время жизни сессии — вечное** (живёт до logout или до перезапуска бэкенда,
  т.к. хранится в памяти процесса). После рестарта бэкенда все сессии инвалидны →
  любой admin-запрос вернёт `401` → редирект на логин.
- **Тела ошибок — это `text/html`, не JSON** (короткий текст вроде `unauthorized`).
  Ориентируйся на **HTTP-статус**, не парси тело ошибки как JSON.
- **Глобальный обработчик `401`:** на любом admin-эндпоинте `401` означает
  «нет/протухла сессия» → очистить состояние и показать экран логина.

> ⚠️ Хост `www.ratzek` — **только HTTP (TLS на нём нет)**, поэтому в конфиге бэкенда
> `cookie_secure: false` (с флагом `Secure` браузер не отдавал бы куку по http). Если
> позже появится HTTPS — флаг переключат на `true`; для фронта это прозрачно.

## 1. Аутентификация

### POST `/api/v1/admin/login`
Вход. Тело:
```json
{ "login": "admin", "password": "••••••" }
```
Ответы:
| Статус | Когда | Тело |
|---|---|---|
| `200` | успех — сервер ставит `Set-Cookie: admin_session=…; HttpOnly; SameSite=Strict; Path=/` | `{ "login": "admin" }` |
| `401` | неверный логин/пароль (или админ не сконфигурирован) | — |
| `400` | тело не распарсилось | — |

После `200` кука уже в браузере — отдельных действий не требуется.

### POST `/api/v1/admin/logout`
Выход. Тело не нужно. Всегда `200`; сервер удаляет сессию и шлёт `Set-Cookie`,
обнуляющий куку. После этого `/me` вернёт `401`.

### GET `/api/v1/admin/me`
Проверка текущей сессии (используй при загрузке приложения и для guard'ов роутов).
| Статус | Тело |
|---|---|
| `200` | `{ "login": "admin" }` |
| `401` | нет/невалидная сессия |

**Рекомендуемый флоу:**
1. На старте → `GET /me`. `200` → показываем админку; `401` → экран логина.
2. Логин → `POST /login` → при `200` повторяем `GET /me` (или сразу переходим внутрь).
3. Logout → `POST /logout` → редирект на логин.

## 2. CRUD безлимитных клиентов

Все эндпоинты требуют валидной сессии (иначе `401`).

**Сущность `UnlimitedClient`:**
```jsonc
{
  "name": "evgenii-phone",      // идентификатор (= имя dhcp-брони), уникален
  "mac": "aa:bb:cc:dd:ee:ff",   // выводится сервером из аренды; в запросе НЕ шлём
  "ip": "10.11.5.50",           // фиксированный IP (безлимит)
  "comment": "Евгений, личный"  // опционально; поле может отсутствовать, если пусто
}
```

**Ограничения полей (валидируй на клиенте до отправки):**
- `name` — slug по маске `^[a-z0-9-]{1,63}$` (строчные латинские буквы, цифры, дефис).
  Должно быть уникальным.
- `ip` — должен входить в подсеть **`10.11.5.0/24`** *и* иметь **активную DHCP-аренду**
  (устройство должно быть подключено/известно). MAC берётся из этой аренды.
- `comment` — необязательный, ≤ 256 символов.
- `mac` — клиент **не передаёт**; возвращается в ответах.

### GET `/api/v1/admin/unlimited-clients`
Список клиентов. **Серверная пагинация/сортировка/фильтрация** (см. §10).
→ `200`, тело: массив `UnlimitedClient`, заголовок `X-Total-Count`.
```json
[
  { "name": "andrey-phone", "mac": "7c:fd:6b:a4:b2:2f", "ip": "10.11.5.236" },
  { "name": "anya-phone",   "mac": "14:2d:4d:e2:1f:57", "ip": "10.11.5.225" }
]
```
- `sort` ∈ `{name, ip, mac, comment}` (дефолт `name asc`); `q` — substring по `name/ip/mac/comment`.
- Параметры пагинации, `X-Total-Count` и режим «вернуть всё» — см. §10. `ip` сортируется численно.

### GET `/api/v1/admin/unlimited-clients/{name}`
Один клиент.
| Статус | Тело |
|---|---|
| `200` | `UnlimitedClient` |
| `404` | имя не найдено |

### POST `/api/v1/admin/unlimited-clients`
Создать. Тело:
```json
{ "name": "evgenii-phone", "ip": "10.11.5.50", "comment": "необязательно" }
```
| Статус | Когда |
|---|---|
| `201` | создан; тело — `UnlimitedClient` (с уже подставленным `mac`) |
| `400` | невалидно: имя не slug; `ip` вне `10.11.5.0/24`; для `ip` нет **активной** аренды; тело не распарсилось |
| `409` | такое `name` или `ip` уже под управлением |
| `500` | ошибка применения dhcp-брони (см. §6); операция не применилась, можно повторить |

### DELETE `/api/v1/admin/unlimited-clients/{name}`
Удалить.
| Статус | Когда |
|---|---|
| `204` | удалён (тела нет) |
| `404` | имя не найдено |
| `500` | ошибка снятия побочных эффектов — можно повторить |

### PATCH `/api/v1/admin/unlimited-clients/{name}`
Изменить **только `comment`** (смена `ip`/`name` — через delete+create). Без dhcp-брони/ipset
side-effects. Тело:
```json
{ "comment": "новый комментарий" }   // или { "comment": null } чтобы очистить
```
| Статус | Когда |
|---|---|
| `200` | обновлено; тело — `UnlimitedClient` |
| `404` | имя не найдено |
| `400` | `comment` > 256 символов / тело не распарсилось |

## 3. DHCP-аренды (`GET /api/v1/dhcp`)

Список всех текущих аренд из `dhcpd.leases`. Нужен и как самостоятельный раздел
админки («устройства в сети»), и для формы выбора IP при создании безлимита.

> **Эндпоинт публичный** (без сессии). Тем не менее, в админке держи его за общим
> guard'ом, чтобы не светить список устройств неавторизованным.

→ `200`, массив записей:
```jsonc
{
  "ip": "10.11.5.50",
  "mac": "aa:bb:cc:dd:ee:ff",        // может быть null
  "hostname": "Pixel-7",             // может быть null
  "client_hostname": "android-...",  // может быть null
  "vendor_class_identifier": null,   // может быть null
  "starts": "Tuesday 2026/06/28 ...",// строка даты или null
  "ends":   "Tuesday 2026/06/28 ...",// строка даты или null
  "acl":    { "ip": "10.11.5.50", "timeout": { "secs": 3600, "nanos": 0 }, "bytes": null } | null,
  "shaper": { "ip": "10.11.5.50", "timeout": { "secs": 600,  "nanos": 0 }, "bytes": 12345 } | null
}
```
Поля `acl`/`shaper` — запись соответствующего ipset (объект `{ip, timeout?, bytes?}`,
`timeout` сериализуется как `{secs, nanos}` или `null`) либо `null`, если IP в наборе нет.
Наличие `acl` = устройство имеет доступ в интернет; наличие `shaper` = устройство шейпится.

**Пагинация/сортировка/фильтрация** (см. §10) + заголовок `X-Total-Count`:
- `sort` ∈ `{ip, mac, hostname, ends}` (дефолт `ip asc`); `q` — substring по `ip/mac/hostname/client_hostname`.
- **Доменные фильтры**: `ip_prefix=<str>` (только аренды с `ip`, начинающимся на строку),
  `has_mac=true|false` (только с непустым/пустым `mac`).

**UX-подсказка для формы создания безлимита:** запроси `?ip_prefix=10.11.5.&has_mac=true`
(режим «вернуть всё»), покажи `ip` + `hostname`/`mac`, дай ввести `name`/`comment`, `POST`
(MAC сервер подставит сам).

## 4. Состояние системы (`GET /api/v1/admin/status` — JSON, и `/metrics` — Prometheus)

### GET `/api/v1/admin/status` (для дашборда)
Под `AuthSession`. JSON-снимок состояния системы. → `200`:
```jsonc
{
  "internet_available": true,
  "speedtest": { "download": 0.0, "upload": 0.0, "ping": 0.0 } | null, // null если не измерялось
  "isp_balance": 0.0 | null,
  "last_tariff_update_secs": -123 | null,   // секунд с обновления тарифа (отрицательное)
  "clients_in_acl": 48,                     // клиентов с доступом
  "clients_in_shaper": 24,                  // клиентов под шейпингом
  "dhcp_leases": { "free": 0, "active": 0, "abandoned": 0 }
}
```
Опциональные поля сериализуются как `null`. Строй дашборд на этом эндпоинте.

> **Производительность:** каждый вызов читает `dhcpd.leases` (~4k строк) + 2 ipset.
> Поллить не чаще ~30–60 сек.

### GET `/metrics` (Prometheus, для мониторинга)
Публичный, **текстовый `text/plain`**. Те же данные в Prometheus exposition. Метрики (gauge):
| Метрика | Смысл |
|---|---|
| `ratzek_internet_available` | есть ли интернет наружу: `1`/`0` |
| `ratzek_speedtest_download` | скорость скачивания (последний speedtest) |
| `ratzek_speedtest_upload` | скорость отдачи |
| `ratzek_speedtest_ping` | пинг |
| `ratzek_isp_balance` | баланс у мобильного оператора |
| `ratzek_last_tariff_update` | секунд с последнего обновления тарифа (отрицательное) |
| `ratzek_clients_in_acl` | сколько клиентов имеют доступ (в ACL ipset) |
| `ratzek_clients_in_shaper` | сколько клиентов шейпятся |
| `ratzek_dhcp_leases_free` / `_active` / `_abandoned` | счётчики DHCP-аренд по состоянию |

`speedtest_*`, `isp_balance`, `last_tariff_update` присутствуют только если данные
уже собраны (иначе строки метрики просто нет). Формат — стандартный Prometheus
exposition (`# HELP`, `# TYPE`, затем `имя значение`).

**Дашборд админки** может строиться на этих данных: статус интернета, скорость,
баланс оператора, число активных клиентов, занятость DHCP-пула.

## 5. Клиентские эндпоинты (для понимания, в админке не нужны)

Это API **captive-portal страницы** для конечного устройства, а не для админа.
Оба работают **по IP вызывающего** (`X-Real-IP` / peer-адрес) и требуют, чтобы
вызывающий был DHCP-клиентом сети. В админке использовать не нужно (они оценивали
бы IP самого админа). Документированы для полноты.

### GET `/api/v1/client`
Статус доступа для вызывающего устройства. → `200` JSON `ServiceInfo`:
```jsonc
{
  "internet_connection_status":
      "Inactive"                       // не зарегистрирован
    | "ClientBlacklisted"              // MAC в чёрном списке
    | { "Connected": {                 // активен
        "bytes_sent": 12345,
        "bytes_unlimited_limit": 1073741824,
        "shaper_reset_secs": 600,
        "connection_forget_secs": 3600
      } },
  "internet_clients_connected": 24,    // всего активных клиентов
  "is_internet_available": true
}
```

### POST `/api/v1/client`
Саморегистрация вызывающего устройства (кнопка «подключиться» на портале):
добавляет его IP в нужные ipset'ы. → `200` (пустое тело) при успехе.

> Существующий фронт портала (репозиторий `ratzek-services-frontend`) использует
> именно эти два эндпоинта. Админка — отдельное приложение и их не трогает.

## 6. Как применяются dhcp-брони

Бэкенд управляет dhcpd `host`-бронями через **include-файл, генерируемый из стора**
(секция `dhcp_reservations` в конфиге): на каждый `POST`/`DELETE` он перегенерирует
файл, валидирует `dhcpd -t` и **перезапускает dhcpd** (~1–2с; клиенты держат текущие
аренды). OMAPI/omshell больше не используется (оказался ненадёжен на этом железе).

Для фронта это прозрачно, но учитывай:
- `POST`/`DELETE` могут отвечать с небольшой задержкой (рестарт dhcpd под капотом).
- **`500` на `POST`/`DELETE`** = ошибка применения брони (например, `dhcpd -t` не прошёл) —
  показывай «не удалось применить, попробуйте ещё раз», операция **не применилась**
  (бэкенд откатывает файл). Безопасно повторить.
- Если секция `dhcp_reservations` в конфиге отсутствует — `POST` сразу `500` (управление
  бронями выключено); на проде она настроена.

## 7. Примеры (curl)

```bash
BASE=http://www.ratzek        # прод (за nginx). Локально/напрямую: http://127.0.0.1:8888

# Логин (сохраняем куку в jar)
curl -i -c jar -X POST $BASE/api/v1/admin/login \
  -H 'content-type: application/json' \
  -d '{"login":"admin","password":"••••••"}'

# Кто я
curl -b jar $BASE/api/v1/admin/me                       # {"login":"admin"}

# Список
curl -b jar $BASE/api/v1/admin/unlimited-clients

# Создать
curl -i -b jar -X POST $BASE/api/v1/admin/unlimited-clients \
  -H 'content-type: application/json' \
  -d '{"name":"evgenii-phone","ip":"10.11.5.50","comment":"личный"}'

# Изменить комментарий
curl -i -b jar -X PATCH $BASE/api/v1/admin/unlimited-clients/evgenii-phone \
  -H 'content-type: application/json' -d '{"comment":"обновлённый"}'

# Удалить
curl -i -b jar -X DELETE $BASE/api/v1/admin/unlimited-clients/evgenii-phone

# Статус системы (дашборд)
curl -b jar $BASE/api/v1/admin/status

# Логаут
curl -b jar -X POST $BASE/api/v1/admin/logout
```

## 8. Чек-лист для фронта

- [ ] Сборка SPA с base path `/ap-admin/`, деплой в `/var/www/ratzek-ap-admin/`
      (URL: `http://www.ratzek/ap-admin/`).
- [ ] `fetch` со `credentials: 'include'`, API по абсолютным `/api/v1/...` (тот же origin).
- [ ] Глобальный перехват `401` → экран логина.
- [ ] Guard роутов через `GET /me` при загрузке.
- [ ] Клиентская валидация `name` (slug), `ip` (в `10.11.5.0/24`), `comment` (≤256).
- [ ] Форма создания: выбор IP из активных аренд (`GET /api/v1/dhcp`, фильтр `10.11.5.*`).
- [ ] Обработка `409` (дубликат name/ip) и `400` (нет активной аренды / невалидно).
- [ ] Обработка `500` на `POST`/`DELETE` (ошибка применения брони; повторить — см. §6).
- [ ] Статусы ошибок читаем по HTTP-коду, тело ошибки — текст, не JSON.

## 9. Сводка эндпоинтов и известные пробелы

| Метод/путь | Назначение | Сессия | В админке |
|---|---|---|---|
| `POST /api/v1/admin/login` | вход | — | да |
| `POST /api/v1/admin/logout` | выход | да | да |
| `GET /api/v1/admin/me` | проверка сессии | да | да |
| `GET /api/v1/admin/unlimited-clients` | список безлимитов | да | да |
| `GET /api/v1/admin/unlimited-clients/{name}` | один безлимит | да | да |
| `POST /api/v1/admin/unlimited-clients` | создать безлимит | да | да |
| `PATCH /api/v1/admin/unlimited-clients/{name}` | изменить `comment` | да | да |
| `DELETE /api/v1/admin/unlimited-clients/{name}` | удалить безлимит | да | да |
| `GET /api/v1/admin/status` | JSON-статус системы (дашборд) | да | да |
| `GET /api/v1/dhcp` | список DHCP-аренд | нет* | да (устройства / выбор IP) |
| `GET /metrics` | метрики Prometheus | нет | мониторинг |
| `GET /api/v1/client` | статус доступа вызывающего | нет | нет (клиентский) |
| `POST /api/v1/client` | саморегистрация вызывающего | нет | нет (клиентский) |

\* публичный, но прячь за общим guard'ом админки.

**Безопасность:** хост `www.ratzek` — **только HTTP (нет TLS)** и `GET /api/v1/dhcp`
публичный, поэтому админку держать в **trusted LAN** (staff-сеть, не guest).

**Известные пробелы (вне текущего scope):**
- **`blacklisted_macs` правится только в конфиге** — API управления чёрным списком нет.
- **Смена `ip`/`name` безлимит-клиента** — только delete+create (PATCH меняет лишь `comment`).
- **Применение броней** идёт через генерируемый include-файл + рестарт dhcpd (см. §6);
  `POST`/`DELETE` полноценны, отдельных оговорок для фронта нет.

## 10. Списки: пагинация, сортировка, фильтрация

Применяется к `GET /api/v1/admin/unlimited-clients` и `GET /api/v1/dhcp`.
Реализует контракт `ratzek-services-frontend-admin/doc/backend-pagination-spec.md`.

**Query-параметры** (все опциональны, snake_case):

| Параметр | Тип | Дефолт | Описание |
|---|---|---|---|
| `page` | int ≥1 | `1` | Номер страницы (1-based) |
| `per_page` | int 1..200 | `25` | Размер страницы |
| `sort` | string | per-endpoint | Поле сортировки (из белого списка эндпоинта) |
| `order` | `asc`\|`desc` | `asc` | Направление |
| `q` | string | — | Substring-фильтр (case-insensitive) |

- **Порядок**: фильтрация (`q` + доменные) → сортировка → пагинация.
- **Режим «вернуть всё»**: если **не передан ни `page`, ни `per_page`** — полный
  отфильтрованный/отсортированный список (`q`/`sort` всё равно применяются).
- **`X-Total-Count`** — заголовок, **всегда присутствует**: число записей после
  фильтрации, **до** пагинации (для UI «25 из 137»). Same-origin → читается без CORS.
- Сортировка **стабильна** (вторичный ключ: `name` для клиентов, `ip` для аренд);
  поля-`ip` сортируются численно.

**Валидация → `400`** (тело — текст, ориентир HTTP-код): `page<1`/нечисло;
`per_page<1`/`>200`/нечисло; `sort` вне белого списка; `order` ∉ {asc,desc};
`has_mac` ∉ {true,false,1,0}. `page` за пределами данных → `200` + **пустой массив**
+ корректный `X-Total-Count` (не `404`).

```bash
# первая страница, по 25, сортировка по ip
curl -i -b jar "$BASE/api/v1/admin/unlimited-clients?sort=ip&order=asc&page=1&per_page=25"
# → 200, X-Total-Count: 137, массив из ≤25 UnlimitedClient
```

## 11. Метаданные устройств (метрики, трафик)

Если на бэкенде включён `device_metrics`, периодический сэмплер (раз в 5 мин) пишет в
SQLite первое/последнее появление устройства в сети и накопленный трафик (из
dhcpd-аренд + ipset-счётчиков). Это обогащает листинги и добавляет новый эндпоинт.
Метрики **best-effort**: если стор недоступен — соответствующие поля `null`, не ошибка.

### Доп. поля `UnlimitedClient` (в `GET /unlimited-clients` и `/{name}`)
Объект клиента дополняется (все поля **аддитивны**, могут быть `null`):
```jsonc
{
  "name": "...", "mac": "...", "ip": "...", "comment": "...",
  "created_at": 1751200000,   // unix-сек, когда запись добавлена (или null у старых)
  "updated_at": 1751200000,   // unix-сек последнего изменения
  "online": true,             // есть активная аренда на этом IP сейчас
  "stale_reservation": false, // активная аренда на IP принадлежит ДРУГОМУ MAC (конфликт брони)
  "hostname": "Pixel-7",      // имя/вендор из текущей аренды или из метрик
  "vendor": null,
  "first_seen": 1750000000,   // метрики: первое наблюдение устройства (по MAC)
  "last_seen": 1751200000,    // метрики: последнее наблюдение
  "bytes_total": 5368709120,  // накопленный трафик (приблизительный)
  "bytes_today": 104857600,
  "bytes_7d": 1073741824
}
```
> `created_at`/`updated_at` доступны и без `device_metrics` (хранятся в сторе). Остальные
> метрики — только когда `device_metrics` включён и сэмплер успел отработать.

### `GET /api/v1/dhcp` — доп. поля
К записи аренды добавлены `last_seen` (cltt — когда устройство последний раз общалось с
dhcpd) и `tstp`. Счётчики `packets`/`bytes` доступны в объектах `acl`/`shaper`.

### `GET /api/v1/admin/devices` (требует сессии)
Единый инвентарь всех известных устройств (по MAC). Пагинация/сортировка/фильтрация — как
в §10. **`sort`** ∈ `{last_seen, first_seen, bytes_total, mac, last_ip}` (дефолт
`last_seen desc` имеет смысл — но дефолт-порядок `asc`; передавайте `order=desc`); **`q`** —
substring по `mac`/`last_ip`/`hostname`/IP из истории. Заголовок `X-Total-Count`.
```jsonc
{
  "mac": "aa:bb:cc:dd:ee:ff",
  "last_ip": "10.11.5.50",
  "ips": ["10.11.4.50", "10.11.5.50"],   // история IP (роуминг)
  "hostname": "Pixel-7", "vendor": null,
  "first_seen": 1750000000, "last_seen": 1751200000,
  "bytes_total": 5368709120, "bytes_today": 104857600, "bytes_7d": 1073741824,
  "online": true,
  "is_unlimited": true                   // есть ли в сторе безлимитов
}
```
Если `device_metrics` выключен — эндпоинт отдаёт `200` с пустым массивом.

> **Точность трафика:** счётчики ipset кумулятивны и сбрасываются при пересоздании записи;
> трафик между сэмплами в момент пересоздания может потеряться, `first_seen` — первое
> наблюдение сэмплером (не историческое). Подходит для оценки, не для биллинга.
